## The problem
A security product built from multiple cooperating agents — some that probe for threats,
some that respond, some that protect the data environment. When several agents act on a
user's behalf, the hardest problem isn't any single screen. It's whether the *whole system*
speaks one coherent language: do the core terms mean the same thing everywhere, and is it
clear how they relate to each other?

## My role
I own the content strategy for how the system communicates — not string by string, but as a
system:

- **Narrative & storyline** — what this product *is*, told so a user can hold it in their head.
- **Terminology & object guides** — a defined entry for every core term, written once and
  reused.
- **The relationships between terms** — the part people skip: how each concept connects to the
  others inside the system, so meaning holds across agents and surfaces instead of drifting.

## The craft move
I author these as **markdown skill files** — structured strategy documents that design,
composition, and engineering pull from directly. Instead of content arriving late to decorate
a finished flow, the language *is* part of the build inputs. That raises both the pace and the
quality of the prototypes we test.

## Why it matters
This is content design as system architecture: define the vocabulary and its internal logic
once, and every agent, screen, and teammate inherits the same clear mental model. It's the
clearest proof of my belief that clarity is a system, not a sentence.
